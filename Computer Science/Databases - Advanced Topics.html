Indexes

What is an index of a database?

  • Primary mechanism to get improved performance on a database
  • Presistent data structures, stored in database
  • Make queries more efficient by skipping the scanning of the full table

Utility

  • Index = difference between full table scans and imeediate location of tuples (rows)
    • order of magnitude performance differences
  • Underlying data structures
    • Balanced trees (B trees / B+ trees)
      • for both equality and inequality queries
      • O(log n)O(\text{log } n)O(log n) performance
    • Hash tables
      • for attribute = value queries
      • O(1)O(1)O(1) performance if well designed
  • Many DBMS’s build indexes automatically on Primary Key (and somestimes Unqiue) attributes.
  • Implementation: Query planning and optimization

Downsides of indexes

  1. Extra space - minor problem
  2. Index creation - medium, takes time
  3. Index maintainance (upon modification of table) - the most important concern

Benefits of an index depends on

  • Size of table (and possibly layouts)
  • Data distribution
  • Query sv. update load

“Physical” design advisor

Input: Database (statistics) and workload
Output: Recommanded indexes

Query optimizer

Input: Database (statistics), query or update, and existing indexes
Output: Best execution with an estimated cost

SQL Syntax:

Create an index on the attribute A

CREATE INDEX indexName on T(A);

Create n indexes on the attribute A1 to An

CREATE INDEX indexName on T(A1, A2, ..., An);

Create a unique index on the attribute A

CREATE UNIQUE INDEX indexName on T(A);

Delete an index

DROP INDEX indexName;

Transactions

Motivated by two independent requirements:

  • Concurrent database access
  • Resilience to system failure

Concurrent Access (by different applications)

  • Attribute level
  • Tuple level
  • Table level
  • Multi-statement level

Concurrency goals

  • Execute sequence of SQL statements so that they appear to be runing in isolation.
    • Simple solution: execute them in sequence
  • But want to enable concurrency when safe to do so.
    • e.g. Multiprocessor, multithreaded, asyn. I/O

Resilience to System Failures

  • Failure scenarios:
    • Crash during a bulk load of data
    • Crash during table migration
    • Crash during lots of updates
  • System failure goal: Guarantee all-or-nothing execution, regardless of failure

Solution to both Concurrency & Failures: Transactions

A transaction is a squence of one or more SQL operations treated as a unit. SQL standard:

  • Transaction begins automatically on the first SQL statement
  • On “commit” tranaction ends and new one begins
  • Current transaction ends on session termination
  • “Autocommit” turns each statement into transaction

Other properties:

  • Transactions appear to run in isolation
  • If the system fails, each transactions changes are relected with entirely / not at all.

ACID Properties

  • Atomicity - each transaction is "all-or-nothing’, never left half done (logging).
  • Consistency
  • Isolation
  • Durability - If system craches after tranaction commits, all effect of transactions remain in the database (logging).

Serializability

  • Operations may be interleaved, but execution must be equivalent to some sequential (serial) order of all transactions.

Transaction Rollback (= Abort)

  • Undoes partial effects of transactions (Atomicity)
  • Can be system- / client-initiated
  • “Rollback” command only undoes the effects on data itself in the DB. It doesn’t un do things like modifying variables.

Consistency:

  • For each client and each transaction:
    • assume all constraints hold when transaction begins
    • must guarantee all constaints hold when transaction ends
  • Serializability => constraints always hold

Isolation Level

4 types of isolation level:

  • Read Uncommitted
  • Read Committed
  • Repeatable Read
  • Serizlizable (by default)

A “weaker” isolation level reduce overhead and increase concurrency, but at the same time reduce the concurrency guarantee.

Dirty Reads

“Dirty” data item: written by an uncommitted transaction

Isolation Level: READ UNCOMMITTED

  • A transaction may perform dirty reads
  • Example: T1 and T2 run concurrently
--T1:
UPDATE Student SET GPA = (1.1)*GPA WHERE sizeHS > 2500;

concurrent with …

--T2: 
SELECT AVG(GPA) FROM Student;

By default, T1 and T2 are serialized, resulting in an operation order of either {T1, T2} or {T2, T1}.

To increase the concurreny, we can set a “weaker” isolation level for T2 as the follows:

--T1:
UPDATE Student SET GPA = (1.1)*GPA WHERE sizeHS > 2500;

concurrent with …

--T2: 
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT AVG(GPA) FROM Student;

This makes T1 & T2 run concurrently, i.e. T2 computes the average of GPA while T1 is updating the GPA.

Isolation Level: READ COMMITTED

  • A transaction may not perform dirty reads
  • Still doesn’t guarantee global serializability
--T1:
UPDATE Student SET GPA = (1.1)*GPA WHERE sizeHS > 2500;

concurrent with …

--T2: 
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT AVG(GPA) FROM Student;
SELECT MAX(GPA) FROM Student;
  • Possible outcome: AVG(GPA) -> GPA = (1.1)*GPA -> MAX(GPA)

Isolation Level: REPAETABLE READ

  • A transaction may not perform dirty reads
  • An iteam read multiple times cannot change value
  • still doesn’t guarantee global serializability

Example 1:

-- T1:
UPDATE Student SET GPA = (1.1) * GPA;
UPDATE Student SET sizeHS = 1500 WHERE sID = 123;

concurrent with …

--T2:
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT AVG(GPA) FROM Student;
SELECT AVG(sizeHS) FROM Student;

A possible concurrent sequence: SET GPA -> AVG(GPA) -> SET sizeHS -> AVG(sizeHS)

Phantom tuples

  • But a relation can change!

Example 2:

-- T1:
INSERT INTO Student [100 new tuples]

concurrent with …

--T2:
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT AVG(GPA) FROM Student;
SELECT AVG(GPA) FROM Student;

Possible outcome:
The two AVG(GPA) statements return different results. The first execution of AVG(GPA) only locks the existing tuples. The insertions of new tuples between the two executions of AVG(GPA) are allowed.

  • However, deletion of tuples is not allowed!

Example 3:

-- T1:
DELETE FROM Student [100 tuples]

concurrent with …

--T2:
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT AVG(GPA) FROM Student;
SELECT AVG(GPA) FROM Student;

Possible outcome:

  • Two statements return the same value, either the AVG(GPA) before or after the complete deletion.
  • If the first AVG(GPA) is executed before T1, all tuples in Student are read and locked, preventing any deletion before T2 completes.

READ ONLY Transactions

  • Help system optimize performance
  • Independent of isolation level

Example:

SET TRANSACTION READ ONLY;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT AVG(GPA) FROM Student;
SELECT MAX(GPA) FROM Student;

The first line hints the query optimizer that the transaction performs only querying operations.

Isolation Level: Summary

Isolation Level Dirty Read Non-repeatable Read Phantoms
Weakest Read Uncommitted Y Y Y
Read Committed N Y Y
Repeatable Read N N Y
Strongest Serializable N N N
  • Standard Default: Serializable
  • Weaker isolation levels:
    • Increased conccurency + decreased overhead = increased performance
    • Weaker consistency guarantees
    • Some systems, e.g. MySQL and Oracle, have default Repeatable Read
  • Isolation level per transaction and “eye of the beholder”
    • Each transaction’s reads must conform to its isolation level.

Motivation & Overview

What are the differences between contraints and triggers?

(Integrity) constraints

  • constrain allowable database states
  • more a static concept

Triggers

  • monitor database cahnges
  • check condition & initiate actions
  • more a dynamic concept

Integrity Constraints

What is an integrity constraint?

  • Implose restrictions on allowable data, beyond those imposed by structure & type.
  • Examples:
    • 0.0 < GPA <= 4.0
    • enrollment < 75,000
    • decision: ‘Y’, ‘N’, or Null
    • major = ‘CS’ then decision = Null

Why use constraints?

  • Catch data entry errors (during INSERT)
  • Set correctness criteria (during UPDATE)
  • Enforce consistency
    • for data come from differenct sources
  • Tell the DBMS about the data
    • for more efficient storing / query processing

Classification of SQL contraints

  • Non-null constraint
  • Key constraint - for unique values
  • Referential integrity (a.k.a. foreign key)
  • Attribute-based
  • Tuple-based
  • General assertions

Declaration & Enforcement

When do we declare a constraint?

  • with original schema - checked after bulk loading
  • or later - checked on current DB

When is a constraint enforced?

  • check after every “dangerous” modification
    • only those potentially cuasing a constraint to be violated
  • deferred constraint checking - check after every transaction instead

Triggers

What is a SQL trigger?

  • When event occurs, check condition; if true, do action.
  • a.k.a. “Event-Condition-Action Rules”

Examples

  • Event: enrollment > 35,000; Action: reject all applications
  • Event: Insert applicant with GPA > 3.95; Action: accept automatically
  • Event: Update sizeHS > 7,000; Action: raise error

Why use triggers?

  • Move logic from applications to DBMS
    • makes the checking avaliable for mutiple applications interacting with the DBMS
  • To enforce constraints
    • expressiveness
    • constraint “repair” logic - by the “action”

Syntax

CREATE TRIGGER name
BEFORE | AFTER | INSTEAD OF events
[referencing-variables]
[FOR EACH ROW]
WHEN (condition)
action;

Constraints of Several Types

Demo datasets:

Simple college admission database

College(cName, state, enrollment)
Student(sID, sName, GPA, sizeHS)
Apply(sID, cName, major, decision)

Non-null constraint

CREATE TABLE Student(
	sID INT,
	sName TEXT,
	GPA REAL NOT NULL,
	sizeHS INT
);

Key constraint

CREATE TABLE Student(
	sID INT PRIMARY KEY,
	sName TEXT,
	GPA REAL,
	sizeHS INT
);

--s1
INSERT INTO Student (sID, sName, GPA, sizeHS)
	VALUES (123, 'Amy', 3.9, 1000);
--s2
INSERT INTO Student (sID, sName, GPA, sizeHS)
	VALUES (456, 'Bob', 3.6, 1500);
--s3: erroneous
INSERT INTO Student (sID, sName, GPA, sizeHS)
	VALUES (123, 'Charles', 3.3, 2000);

s3 will incur an insert error due to the repeated sID.

Description: Abort due to constraint violation. Column sID is not unique.

The following update operation may also incur the same error. This depends on which tuples are updated first.

UPDATE Student
	SET sID = sID - 111;

This issue can be mitigated by using the deferred constraint checking.

Multiple Keys

CREATE TABLE Student(
	sID INT PRIMARY KEY,
	sName TEXT PRIMARY KEY,
	GPA REAL,
	sizeHS INT
);

Description: Table "Student" has more than one primary key.

In SQL standards, only 1 primary key is allowed for a table. Setting other attributes as UNIQUE to set up multiple keys in a single table.

CREATE TABLE Student(
	sID INT PRIMARY KEY,
	sName TEXT UNIQUE,
	GPA REAL,
	sizeHS INT
);

Multiple-attribute Key

CREATE TABLE College(
	cName TEXT,
	state TEXT,
	enrollment INT,
	PRIMARY KEY (cName, state)
);

--s1
INSERT INTO College (cName, state, enrollment)
	VALUES ('Mason', 'CA', 10000);
--s2
INSERT INTO College (cName, state, enrollment)
	VALUES ('Mason', 'NY', 5000);
--s3: erroneous
INSERT INTO College (cName, state, enrollment)
	VALUES ('Mason', 'CA', 2000);

Description: Abort due to constraint violation. Column cName, state are not unique.

To enable more than 1 attribute keys, use the UNIQUE constraint as before.

Example

CREATE TABLE Apply(
	sID INT,
	cName TEXT,
	major TEXT,
	decision TEXT,
	UNIQUE (sID, cName),
	UNIQUE (sID, major)
);

--s1
INSERT INTO Apply (sID, cName, major, decision)
	VALUES (123, NULL, NULL, 'Y');
--s2
INSERT INTO Apply (sID, cName, major, decision)
	VALUES (123, NULL, NULL, 'N');

In SQL standard, repeated NULL vaules are allowed for attributes declared as unique.

Attribute-based check constraints

We can use CHECK to create constraints on attributes when creating a schema. This catches data-entry errors and data modification errors.

Example

CREATE TABLE Student(
    sID INT,
    sName TEXT,
    GPA REAL CHECK(
        GPA <= 4.0
        AND GPA > 0.0
    ),
    sizeHS INT CHECK(sizeHS < 5000)
);

--s1
INSERT INTO Student (sID, sName, GPA, sizeHS)
	VALUES (123, 'Amy', 3.9, 1000);
--s2: erroneous
INSERT INTO Student (sID, sName, GPA, sizeHS)
	VALUES (456, 'Bob', 4.6, 1500);

Description: Abort due to constraint violation. Constraint failed.

The attribute-based constraint can be used to enforce the non-null constraint syntaxtically.

Example

CREATE TABLE Student(
    sID INT,
    sName TEXT,
    GPA REAL CHECK(GPA IS NOT NULL),
    sizeHS INT
);

Tuple-based constraints

The constraints are checked whenever a tuple is inerted or updated.

Example

CREATE TABLE Apply(
    sID INT,
    cName TEXT,
    major TEXT,
    decision TEXT,
    CHECK (
        decision = 'N'
        OR cName <> 'Stanford'
        OR major <> 'CS'
    )
);

General assertions

Assertions are extremely powerful. It is in the SQL standard. However, it is may not be supported by the system.

Example

Assert the attribute A is always unique.

CREATE ASSERTION Key CHECK (
    (
        SELECT COUNT(DISTINCT A)
        FROM T
    ) = (
        SELECT COUNT(*)
        FROM T
    )
);

Example

Assert the student IDs in the Apply table already exist in Student table.

CREATE ASSERTION ReferentialIntegrity CHECK (
    NOT EXISTS (
        SELECT *
        FROM Apply
        WHERE sID NOT IN (
                SELECT sID
                FROM Student
            )
    )
);

Example

Assert the average GPA of accepted applicants is always greater than 3.0.

CREATE ASSERTION AvgAccept CHECK (
    3.0 < (
        SELECT AVG(GPA)
        FROM Student
        WHERE sID IN (
                SELECT sID
                FROM Apply
                WHERE decision = 'Y'
            )
    )
);
